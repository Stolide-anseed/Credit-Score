# Исследование признаков для Feature Engineering

Файл фиксирует идеи для feature engineering по `data/train.csv`. Это исследовательский документ: признаки ниже не добавлены в inference/preprocessing pipeline и требуют отдельной проверки на кросс-валидации.

## Контекст данных

- Размер датасета: 100 000 строк, 28 исходных колонок.
- Количество клиентов: 12 500.
- Период наблюдений: 8 месяцев, с January по August.
- Целевая переменная: `Credit_Score`.
- Распределение таргета: `Poor` - 29.00%, `Standard` - 53.17%, `Good` - 17.83%.

Для численного анализа `Credit_Score` был временно закодирован как `Poor = 0`, `Standard = 1`, `Good = 2`. Связь кандидатов с таргетом оценивалась через Spearman correlation и mutual information. Это не доказывает причинность и не заменяет валидацию модели, но помогает выбрать сильные гипотезы.

## Главные наблюдения

1. Самые сильные сигналы дают не отдельные финансовые поля, а комбинации долговой нагрузки, ставки, дохода и кредитной истории.
2. `Credit_Mix` и `Payment_of_Min_Amount` остаются сильными категориальными признаками; их нужно аккуратно кодировать, а не удалять.
3. `Type_of_Loan` содержит заметный сигнал как через количество типов кредитов, так и через сам факт отсутствия/наличия кредитов.
4. Месячная история клиента полезна: предыдущий долг, предыдущая задержка и предыдущие просроченные платежи хорошо связаны с текущим `Credit_Score`.
5. Дельты между месяцами оказались слабее абсолютных лаговых значений. Их стоит тестировать, но не добавлять первыми.
6. `Occupation` и `Month` дают слабый самостоятельный сигнал и должны иметь низкий приоритет.

## Приоритетные числовые кандидаты

| Приоритет | Кандидат | Формула | Spearman с таргетом | Почему может работать |
| --- | --- | --- | ---: | --- |
| P1 | `Interest_Debt_Pressure` | `Outstanding_Debt * Interest_Rate / 100` | -0.479 | Объединяет размер долга и риск-прайсинг. Среднее значение: `Poor` 471.8, `Standard` 228.3, `Good` 71.0. |
| P1 | `Credit_Inquiries_per_History_Year` | `Num_Credit_Inquiries / (Credit_History_Years + 1)` | -0.447 | Частые кредитные запросы при короткой истории выглядят как рискованный профиль. |
| P1 | `Credit_Pressure_Utilized_Debt` | `Outstanding_Debt * Credit_Utilization_Ratio / 100` | -0.420 | Прокси использованного долгового давления. |
| P1 | `Debt_to_Annual_Income` | `Outstanding_Debt / Annual_Income` | -0.400 | Нормализует долг на годовой доход. Среднее: `Poor` 0.099, `Standard` 0.052, `Good` 0.023. |
| P1 | `Credit_History_Years` | `Credit_History_Age_Months / 12` | 0.398 | Длина кредитной истории явно растёт от `Poor` к `Good`: 14.2, 19.0, 23.7 лет. |
| P1 | `Debt_to_Monthly_Salary` | `Outstanding_Debt / Monthly_Inhand_Salary` | -0.396 | Нормализует долг на доступный месячный доход. |
| P1 | `Loan_Type_Count` | число типов в `Type_of_Loan` | -0.354 | У `Poor` в среднем 4.76 типа кредита, у `Good` 2.20. |
| P2 | `EMI_to_Monthly_Salary` | `Total_EMI_per_month / Monthly_Inhand_Salary` | -0.282 | Доля регулярных кредитных платежей в зарплате. |
| P2 | `unsecured_loan_count` | количество необеспеченных типов кредитов | -0.279 | Улавливает структуру кредитного портфеля, не только его размер. |
| P2 | `secured_loan_count` | количество обеспеченных типов кредитов | -0.230 | Слабее общего количества кредитов, но может добавлять структуру портфеля. |
| P3 | `Free_Cash_Flow_Proxy` | `Monthly_Inhand_Salary - Total_EMI_per_month - Amount_invested_monthly` | 0.189 | Приближает финансовый запас после платежей и инвестиций. |
| P3 | `Debt_Service_and_Investment_to_Salary` | `(Total_EMI_per_month + Amount_invested_monthly) / Monthly_Inhand_Salary` | -0.168 | Прокси общей нагрузки на месячный доход. |

Рекомендация: начинать с P1-признаков. Они имеют понятную финансовую интерпретацию и сильную одномерную связь с таргетом.

## Категориальные признаки

### `Credit_Mix`

`Credit_Mix` является самым сильным категориальным кандидатом: mutual information 0.150.

| Значение | Poor, % | Standard, % | Good, % | Средний score |
| --- | ---: | ---: | ---: | ---: |
| `Bad` | 60.08 | 38.40 | 1.52 | 0.414 |
| `Standard` | 21.54 | 72.86 | 5.60 | 0.841 |
| `Missing` | 29.06 | 53.00 | 17.94 | 0.889 |
| `Good` | 15.86 | 35.34 | 48.79 | 1.329 |

Гипотеза: кодировать как ordinal-признак `Bad < Standard < Good`, но отдельно сохранить флаг пропуска, потому что `Missing` не совпадает с худшим уровнем.

### `Payment_of_Min_Amount`

Mutual information 0.105. Признак хорошо разделяет классы.

| Значение | Poor, % | Standard, % | Good, % | Средний score |
| --- | ---: | ---: | ---: | ---: |
| `Yes` | 39.57 | 56.77 | 3.65 | 0.641 |
| `NM` | 29.30 | 52.00 | 18.70 | 0.894 |
| `No` | 13.38 | 48.29 | 38.33 | 1.250 |

Гипотеза: сделать ordinal-кодирование `Yes < NM < No` и проверить отдельно one-hot. `Yes` выглядит как сильный риск-фактор.

### `Payment_Behaviour`

Сам исходный признак слабее (`mutual_info` около 0.007), но его можно разложить:

- `Payment_Spend_Level`: `High`, `Low`, `Missing`.
- `Payment_Value_Size`: `Small`, `Medium`, `Large`, `Missing`.

`High_spent` и `Large_value_payments` связаны с более высоким средним score, но эффект умеренный. Эти признаки лучше добавлять после P1/P2.

## `Type_of_Loan`

`Type_of_Loan` лучше не оставлять одной длинной строкой. Полезные варианты:

1. `Loan_Type_Count` - количество типов кредитов.
2. One-hot флаги по каждому типу кредита.
3. Группировка в `secured_loan_count` и `unsecured_loan_count`.
4. Флаг `is_missing_Type_of_Loan` как отдельный сигнал.

Связь флагов типов кредитов с таргетом умеренная, но стабильная: наличие каждого типа кредита снижает средний score примерно на 0.20 пункта.

| Флаг | Доля строк, % | Средний score если есть | Средний score если нет | Разница |
| --- | ---: | ---: | ---: | ---: |
| `has_Auto_Loan` | 30.56 | 0.735 | 0.956 | -0.222 |
| `has_Personal_Loan` | 31.10 | 0.745 | 0.953 | -0.208 |
| `has_Payday_Loan` | 31.94 | 0.748 | 0.954 | -0.206 |
| `has_Mortgage_Loan` | 31.36 | 0.747 | 0.953 | -0.205 |
| `has_Credit_Builder_Loan` | 31.73 | 0.748 | 0.953 | -0.205 |
| `has_Debt_Consolidation_Loan` | 31.04 | 0.747 | 0.952 | -0.205 |
| `has_Student_Loan` | 31.04 | 0.748 | 0.951 | -0.203 |
| `has_Home_Equity_Loan` | 31.40 | 0.752 | 0.951 | -0.198 |
| `has_Not_Specified` | 31.68 | 0.754 | 0.950 | -0.196 |

Важно: пропуск `Type_of_Loan` сам по себе положительно связан с таргетом (`Spearman` 0.158). Вероятная интерпретация - у клиента нет активного кредита или нет типа кредита. Поэтому этот признак не стоит просто импутировать строкой `Missing`; лучше сделать явный флаг `has_no_reported_loan_type`.

## Missing и anomaly flags

Флаги пропусков и аномалий могут быть полезны не только для очистки, но и как признаки качества/структуры записи.

| Флаг | Spearman | Интерпретация |
| --- | ---: | --- |
| `is_missing_Type_of_Loan` | 0.158 | Пропуск похож на отсутствие активных типов кредита, связан с более высоким score. |
| `is_anomaly_Age` | -0.089 | Некорректный возраст чаще встречается у более плохих записей; использовать осторожно. |
| `missing_marker_count` | 0.067 | Количество пропусков слегка растёт от `Poor` к `Good`, эффект слабый. |
| `is_anomaly_Delay_from_due_date` | 0.065 | Аномальные значения задержки могут отражать отрицательные/нулевые задержки; требует ручной проверки. |

Рекомендация: добавить только несколько осмысленных флагов (`is_missing_Type_of_Loan`, `is_missing_Credit_History_Age`, `is_missing_Monthly_Inhand_Salary`, `anomaly_count`) и проверить, не ухудшают ли они стабильность модели.

## Клиентская история и временные признаки

Данные имеют панельную структуру: у каждого клиента до 8 месячных записей. Это можно использовать, если в inference доступна предыдущая история клиента.

Сильные лаговые кандидаты:

| Кандидат | Spearman | Комментарий |
| --- | ---: | --- |
| `Prev_Delay_from_due_date` | -0.428 | Прошлая задержка почти так же информативна, как текущие риск-признаки. |
| `Prev_Outstanding_Debt` | -0.427 | Предыдущий долг сильно связан с текущим score. |
| `Prev_Num_of_Delayed_Payment` | -0.356 | История просрочек важна. |
| `Prev_Monthly_Balance` | 0.226 | Более высокий предыдущий баланс связан с лучшим score. |
| `Prev_Amount_invested_monthly` | 0.122 | Слабее, но может отражать финансовый запас. |

Дельты между месяцами (`Delta_*`) оказались слабее: корреляции близки к нулю для задержек, utilization и баланса. Их можно тестировать после лаговых абсолютных значений.

Критическое условие: не использовать агрегаты по всей истории клиента, если они включают будущие месяцы относительно предсказываемой записи. Для честной валидации нужны только lag/expanding признаки, рассчитанные по прошлым наблюдениям.

## Признаки с низким приоритетом

- `Occupation`: mutual information около 0.001, самостоятельный сигнал слабый.
- `Month`: mutual information около 0.001. Средний score немного выше в July/August, но эффект мал и может быть артефактом выборки.
- `Cards_per_Bank_Account`, `Loans_per_Bank_Account`: связь слабее P1/P2, но можно проверить как дополнительные ratio-признаки.
- `Investment_to_Monthly_Salary`, `Balance_to_Monthly_Salary`: связь умеренная/слабая; вероятно, они полезнее в сочетании с доходом, долгом и EMI.

## Риски leakage

Нельзя добавлять без отдельной проверки:

- `Customer_ID` как one-hot или target encoding: модель может запомнить клиентов.
- `Name`, `SSN`, `ID`: идентификаторы и персональные поля.
- Любые признаки, рассчитанные по всем месяцам клиента сразу: средний долг за все месяцы, максимальная задержка за весь период, будущие значения относительно текущего месяца.
- Target encoding категорий без nested CV или out-of-fold схемы.
- Импутацию по клиенту, если она использует будущие месяцы.

Для проверки feature engineering лучше использовать `GroupKFold` или holdout по `Customer_ID`, чтобы записи одного клиента не попадали одновременно в train и validation. Для временных признаков дополнительно стоит проверить time-aware split: обучение на ранних месяцах, проверка на более поздних.

## Рекомендуемый план экспериментов

1. Baseline: текущий preprocessing без новых FE.
2. Experiment A: добавить P1 ratio-признаки:
   - `Interest_Debt_Pressure`
   - `Credit_Inquiries_per_History_Year`
   - `Credit_Pressure_Utilized_Debt`
   - `Debt_to_Annual_Income`
   - `Debt_to_Monthly_Salary`
   - `Credit_History_Years`
3. Experiment B: добавить признаки `Type_of_Loan`:
   - `Loan_Type_Count`
   - one-hot loan type flags
   - `secured_loan_count`
   - `unsecured_loan_count`
   - `has_no_reported_loan_type`
4. Experiment C: добавить категориальные переразборы:
   - ordinal `Credit_Mix`
   - ordinal/one-hot `Payment_of_Min_Amount`
   - `Payment_Spend_Level`
   - `Payment_Value_Size`
5. Experiment D: если inference сможет получать историю клиента, добавить lag-признаки:
   - `Prev_Delay_from_due_date`
   - `Prev_Outstanding_Debt`
   - `Prev_Num_of_Delayed_Payment`
   - `Prev_Monthly_Balance`
6. Experiment E: добавить missing/anomaly flags и проверить прирост отдельно.

Метрики сравнения: macro/weighted precision, recall, ROC-AUC OVR, average precision, а также per-class метрики для `Poor`, `Standard`, `Good`. Особое внимание стоит уделить классу `Good`, потому что он самый малочисленный.

## Итоговая рекомендация

Наиболее перспективная первая итерация feature engineering:

```text
Interest_Debt_Pressure
Credit_Inquiries_per_History_Year
Credit_Pressure_Utilized_Debt
Debt_to_Annual_Income
Debt_to_Monthly_Salary
Credit_History_Years
Loan_Type_Count
EMI_to_Monthly_Salary
has_no_reported_loan_type
Credit_Mix_ordinal
Payment_of_Min_Amount_ordinal
```

Эти признаки имеют сильную связь с таргетом, хорошо интерпретируются с точки зрения кредитного риска и не требуют будущей информации. Лаговые признаки стоит рассматривать отдельным блоком только после подтверждения, что inference-сценарий действительно получает исторические записи клиента.
